## Hermes Skill 加载的可见性问题

### 问题
用户问：hermes如何在终端执行时，可以将加载的skill打印出来，目前只看到了skill_view...

### 答案
Hermes 当前**没有**提供在终端执行时实时打印已加载 skills 的机制。

现状分析：
- `skills_list` - 列出所有可用 skills（名称+描述）
- `skill_view` - 查看特定 skill 的详情（需要指定 name）
- **缺失**：执行时追踪/打印已加载 skills 的功能

原因：
1. Skill 加载是隐式自动的 - 系统根据用户输入判断并加载
2. 加载的 skill 内容直接注入系统提示词，没有独立的"已加载列表"
3. 没有设计 skill 加载事件的通知/打印钩子

临时方案：
- 直接问 AI "你现在加载了哪些 skills？"，AI 会根据上下文反推

根本解决方案：
需在 Hermes 内核增加 skill 加载事件机制，类似：
```python
self.emit("skill_loaded", skill_name)  # 可被日志/终端捕获
```

## Hermes skill_view 的加载行为

### 问题
为什么当前窗口没有打印出加载的 skill 名字，只有 "preparing skill_view…"？

### 答案
skill_view 是"静默加载"机制。执行后不会在对话窗口输出 "已加载 skill: XXX" 的确认消息。
- "preparing skill_view…" 只是工具调用的过渡提示
- skill 内容被注入到 system prompt（系统提示词）中，作为内部上下文约束
- 用户侧看不到独立的加载成功消息，只有助手口头回应才能确认当前加载了哪些 skill

### 关键点
- skill_view 加载后，skill 内容进入 system prompt，不输出到对话历史
- 若需显式确认，需依赖助手主动口头汇报当前 loaded skills
- 这与某些 IDE 的"静默加载插件"机制类似

## 【完整会话沉淀】OpenClaw vs Hermes 架构差别 — 费曼学习法实录

### 会话背景
- **学习主题**：OpenClaw 与 Hermes 的核心架构差别
- **学习者背景**：刚开始使用 Hermes，之前有搭建 OpenClaw 环境但未深度使用
- **学习方法**：费曼学习法（意图深挖 → 知识传授 → 质疑验证 → 归档沉淀）

### 用户初始理解
- OpenClaw = 各类 Agent 的管理系统，具备网关，可管理各个终端的通信
- Hermes = 单 Agent，但也具备网关系统

### 第一轮教学（有漏洞）
最初将差别归纳为：
1. OpenClaw 的 Gateway 是"通信路由中枢"，Hermes 的网关是"工具协议适配器"(MCP)
2. OpenClaw 是多 Agent 编排器，Hermes 是单 Agent 深潜执行器
3. OpenClaw 常驻后台，Hermes 按需交互

**漏洞**：暗含"OpenClaw 能接多渠道，Hermes 不能"的错误假设。

### 第一次用户质疑（飞书接入）
用户指出：Hermes 也可以配置飞书应用机器人，在后台运行，通过飞书聊天发送消息，Hermes 执行任务后反馈结果。**感觉和 OpenClaw 接入飞书是一样的**。

**第一次修正**：
- 承认质疑有效：两者接飞书的外部行为确实相似
- 核心差别不在"能不能接渠道"，而在：
  - **Channel 是心脏还是手脚**：OpenClaw 从设计之初就以多渠道为架构核心；Hermes 的渠道是扩展能力
  - **多 Agent 隔离是原生还是临时**：OpenClaw 原生支持多个常驻、隔离的 Agent；Hermes 的 subagent 是任务级临时调用
  - **Gateway 控制平面复杂度**：OpenClaw 有完整的会话/渠道/事件管理；Hermes 的飞书 bot 模式是 Webhook 接收器 + Agent 执行器

### 第二次用户质疑（Agent 管理 Subagent）
用户追问：OpenClaw 是否具备"一个 Agent 管理 Subagent"的概念？

**调研过程**：
- 查阅 OpenClaw 官方文档：`docs/concepts/multi-agent.md` 和 `docs/concepts/delegate-architecture.md`
- 发现 OpenClaw 的"多 Agent"是**平面隔离路由**，不是**层级管理**

**第二次修正**（核心发现）：

| 维度 | OpenClaw 多 Agent | Hermes Subagent |
|---|---|---|
| 关系类型 | 平行隔离（同事/租户） | 层级委派（上下级） |
| 存在目的 | 不同渠道/用户走不同"人格" | 一个复杂任务拆分给专家并行执行 |
| 调用方式 | Gateway 路由，Agent 之间不互相调用 | Parent Agent 显式 spawn subagent，等待结果回传 |
| 生命周期 | 常驻 | 临时，任务结束销毁 |
| 数据隔离 | 完全隔离（各自独立 workspace/session/auth） | 隔离但结果回传父 Agent 整合 |

### 最终结论
> **不是"OpenClaw 能接多渠道，Hermes 不能"，而是"OpenClaw 是专为多渠道+多Agent+网关控制平面设计的系统"；Hermes 是"以深度任务执行为核心的 Agent 框架"，其多渠道接入是扩展能力而非架构内核。**

OpenClaw 的"多 Agent"管理的是"多个平行的 Agent 实例"（多租户隔离），而非"一个 Agent 管理多个 Subagent"（组织架构层级）。

### 认知迭代路径
1. 初始：简单对比"多 vs 单" → 有漏洞
2. 质疑 1：飞书接入相似性 → 修正为"架构原生性之差"
3. 质疑 2：Agent 管理 Subagent 概念 → 核心发现"平行隔离 vs 层级委派"是最深架构分水岭
4. 结果：外行为可以相似，但架构范式完全不同
