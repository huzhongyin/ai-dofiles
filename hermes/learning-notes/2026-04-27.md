## Claude Code Auto Memory 工作机制

### 问题
Claude Code 的 auto memory 是如何工作的？请详细讲解其机制。

### 答案
Claude Code 有两套项目级记忆系统并行的设计：

1. 显式记忆 CLAUDE.md：用户手动编写，存在项目根目录或 ~/.claude/，团队可共享。
2. 隐式记忆 Auto Memory：Claude 自动从对话中提取项目知识，存在 ~/.claude/projects/<project>/memory/，25KB 或 200 行硬限制。

Auto Memory 的触发时机是会话结束或 /compact 后，Claude 回顾会话内容，自动总结架构决策、关键命令、代码约定等。下次进入同一项目时自动注入上下文。

与 CLAUDE.md 的关系：Auto Memory 减少手动维护负担，但重要稳定规则应写入 CLAUDE.md。Auto Memory 不进入 git，属于个人笔记。

### 关键点
- 存储路径: ~/.claude/projects/<project>/memory/
- 硬限制: 25KB 或 200 行（先到者为准）
- 加载方式: 进入项目目录时自动注入系统提示
- 与 CLAUDE.md 是互补关系，非替代
- 官方未公开底层摘要算法、去重机制、项目标识生成规则、淘汰策略等细节
- 快速记忆前缀 `#` 可能进入 CLAUDE.md 或 Auto Memory，落盘位置未明确区分

## Hermes Memory 与 Claude Code Auto Memory 对比

### 问题
Hermes 的 Memory 记忆机制与 Claude Code 的 Auto Memory 有何不同？

### 答案
基于 Hermes 源码（memory_provider.py + memory_manager.py）与 Claude Code 官方文档的精确对比：

1. 架构：Hermes 是插件化抽象基类 MemoryProvider，支持内置 + 1 个外部后端（Honcho/Mem0/Hindsight 等）。Claude Code 是单一内置封闭机制，无扩展点。

2. 写入触发：Hermes 通过显式 memory 工具（add/replace/remove）写入，精确可控。Claude Code 是会话结束后自动摘要提取，隐式不可控。

3. 召回机制：Hermes 每轮 API 调用前执行 prefetch()，用 <memory-context> fence 包裹注入，标注 "NOT new user input"。Claude Code 是进入项目时静态全量加载到系统提示，无动态召回。

4. 同步时机：Hermes sync_turn() 每轮结束后同步。Claude Code 推断为会话边界触发。

5. 存储模型：Hermes 全局 MEMORY.md + USER.md（跨项目共享）。Claude Code 项目级隔离。

6. 大小限制：Hermes 无硬限制（受系统提示长度间接约束）。Claude Code 25KB/200行硬限制。

### 关键点
- Hermes: 显式、插件化、每轮动态召回 → 工程师工具
- Claude Code: 隐式、单体、静态全量加载 → 消费者自动化
- Hermes prefetch() 有 <memory-context> fence 防止误认为用户输入
- Hermes 支持丰富生命周期钩子：on_pre_compress、on_delegation、on_session_end
- 未确认：Claude Code 摘要算法、去重策略、精确 sync 时机
- 未确认：Hermes 的 BuiltinMemoryProvider 具体实现（源码中未找到该类定义）

## Hermes 会话启动流程与记忆机制深度解析

### 问题
1. Hermes 只有 MEMORY.md 和 USER.md 吗？历史会话习惯保存在哪里？
2. 只放在这两个里面会不会上下文爆炸？
3. prefetch() 到底做了哪些事情？
4. 一个会话开始时主要做了哪些工作？

### 答案

**1. 存储架构**
内置记忆确实只有 MEMORY.md + USER.md，但历史会话有多套并行存储：
- SQLite Session DB：完整对话历史 + FTS5 全文搜索（session_search 工具查询）
- Session Log JSON：单会话轨迹（~/.hermes/sessions/session_<id>.json）
- 外部 Provider：Honcho/Mem0/Hindsight 等插件
- Learning Notes：~/.hermes/learning-notes/（独立系统）
- Context Compressor 摘要：仅存于当前会话消息列表，不落盘到 MEMORY.md

**2. 上下文爆炸防护**
三层防护：
- 文件级硬限制：MEMORY.md 2200 字符，USER.md 1375 字符，超限拒绝写入
- Frozen Snapshot：会话启动时捕获快照，整个会话系统提示不变，保护 prefix cache
- ContextCompressor：超过 threshold（默认 50% 上下文窗口）时触发结构化摘要，替换中间轮次

**3. prefetch() 机制**
内置记忆不走 prefetch()！内置记忆通过 format_for_system_prompt() 在 _build_system_prompt() 阶段静态注入。
prefetch() 是 MemoryProvider ABC 中给外部后端用的动态召回接口，每轮 API 调用前执行，结果用 <memory-context> fence 包裹注入用户消息。

**4. 会话启动流程**
__init__ 阶段：生成 session_id → 创建 SQLite 会话 → 初始化 CheckpointManager → 读取 config → MemoryStore.load_from_disk() → 初始化 MemoryManager → 初始化 ContextCompressor → 发现 skills → 工具集注册

run_conversation 阶段：检查 _cached_system_prompt → 新会话则 _build_system_prompt() 构建（共 12 层） → 触发 on_session_start hook → 保存到 SQLite → Preflight Compression（最多 3 轮） → pre_llm_call hook → 进入主循环

### 关键点
- 内置记忆是显式、精简、有限的；历史对话是原始、完整、可搜索的（session_search）
- Frozen Snapshot 模式是 Hermes 记忆设计的核心：系统提示稳定 = prefix cache 命中 = 成本低
- _build_system_prompt() 有 12 层组装顺序，记忆在第 5-6 层（MEMORY.md → USER.md）
- 压缩算法保护 head + tail，摘要 middle，叠代更新时保留 _previous_summary
- 会话连续时重用 SQLite 中存储的 system_prompt，确保 Anthropic cache 前缀一致

## session_search 工具工作机制

### 问题
session_search 查 SQLite Session DB 的作用是什么，是查所有 session 吗？

### 答案
session_search 是跨会话原始对话的全文检索工具，不是查询 session 元数据列表。

查询范围：把扫描所有历史会话的消息，但排除：
1. 当前会话 lineage（当前 + 父/子会话）
2. source="tool" 的隐藏会话

工作流：
1. FTS5 MATCH 查询 messages_fts 虚拟表，BM25 rank 排序，默认取50条消息
2. 按 session 去重，将子/委托会话归并到父会话
3. 加载每个会话的完整对话，围绕查询词截断到 100k 字符
4. 辅助 LLM 并行生成聚焦摘要（默认3个会话，上限5个）
5. 返回每会话的结构化摘要 + 元数据

无 query 时进入浏览模式：list_sessions_rich 返回最近会话的标题/时间/预览，零 LLM 成本。

### 关键点
- FTS5 虚拟表 messages_fts 同步 messages 表的 content
- 排序依据是 BM25 rank，不是时间
- 子会话（委托/压缩产生）通过 parent_session_id 链追溯回父会话
- 截断策略：短语匹配 → 200字符共现 → 单词位置 → 最优5%覆盖窗口
- 辅助 LLM 摘要要求过去时态、保留具体命令/路径/错误信息
- 未确认：DEFAULT_DB_PATH 精确路径、FTS5 触发器同步机制、辅助 LLM 的精确模型路由逻辑

## Hermes Skill 匹配机制

### 问题
有人说 Hermes 匹配 skill 的规则是根据 FTS5 全文检索加语义理解来匹配的，这句话对吗？

### 答案
不对。Hermes 的 skill 匹配没有外部 FTS5 全文检索引擎，也没有独立的语义检索模块。

实际机制是：
1. 所有可用 skills 的元数据（名称、描述、标签、类别）在每次对话启动时完整注入系统提示词。
2. 由大模型自身根据用户输入的语义，从这个完整列表里扫描并选择最相关的 skills。
3. 这是一个纯模型端的语义推理过程，没有外部检索架构参与。

所以"FTS5 + 语义理解"的说法把机制说复杂了，混淆了"模型内建推理"和"外部检索系统"两个概念。

### 关键点
- 无 FTS5 倒排索引，无向量数据库，无独立语义检索模块
- Skills 列表全量注入 system prompt，模型自匹配
- 若听到 skills.db / FTS5 相关说法，可能是社区分支或个人魔改版本

## Hermes delegate_task 机制 vs Claude Code 内部路由规则

### 问题
Hermes 的子Agent委派是Agent自主决定何时分发、怎么汇总结果的，这个是如何做到的呢？和 Claude Code 的内部规则（写文档用 haiku、推理用 opus等）有什么区别？

### 答案

**Hermes delegate_task 的实际机制**

它不是一个"独立的自主调度器"，而是主Agent（当前模型）主动调用的工具。流程是：

1. 主Agent在推理过程中，根据 system prompt 对 `delegate_task` 的描述，判断任务是否适合拆分出去。
2. 如果是，发起一次工具调用，带上 goal、context、toolsets，可选 acp_command（如 `claude --acp --stdio`）。
3. 系统层收到后启动一个完全隔离的新会话——新的 conversation、新的终端 session、独立工作目录。子Agent只能看到传入的 context。
4. 子Agent执行完毕后，只有最终摘要返回给主Agent。中间过程对主Agent不可见，避免上下文污染。
5. 主Agent拿到结果后决定下一步。

所以"自主决定何时分发"有一半对：决策权确实在Agent手里，但它不是通过独立调度模块或硬编码规则实现的，而是模型在 tool-use 层面对工具的自主选择。

**与 Claude Code 内部规则的区别**

两者不在同一个抽象层：

| 维度 | Hermes delegate_task | Claude Code 内部规则 |
|------|----------------------|--------------------------|
| 决策层 | 模型自身 tool-use 推理 | 应用层硬编码的启发式规则 |
| 隔离方式 | 启动独立子进程/会话，完全隔离 | 同一进程内切换模型，上下文连续 |
| 可配置性 | 主Agent/用户显式指定 | 用户无感知，内部自动分发 |
| 模型切换 | acp_command 可换完全不同后端 | 同一生态内切换（haiku/sonnet/opus） |
| 适用场景 | 并行任务、上下文防污染、跨工具集隔离 | 成本优化 |

**本质区别**：Claude Code 是"同一厨房里换厨师"，任务在一个上下文里，换不同水平的厨师来做。Hermes 的 delegate_task 是"把订单分发给隔壁独立厨房"，子Agent只能看到订单上写的信息，只有最终成品交换。

**常见误解**：有人觉得 Hermes 的"子Agent"是一个更聪明的调度器在自动拆分任务。其实不是——拆分的决策者是当前主模型，delegate_task 只是提供了一个"启动隔离环境并执行"的系统原语。

### 关键点
- delegate_task 是工具层原语，不是独立调度Agent
- 子Agent 完全隔离，只最终摘要返回
- Claude Code 路由是应用层硬编码规则，Hermes 是模型层 tool-use 决策
- 两者目标不同：前者为隔离与并行，后者为成本与性能

## Hermes delegate_task vs Claude Code subagent 修正笔记

### 问题
用户质疑：Claude Code 分配子 agent 不是隔离上下文的吗？请结合 Anthropic 官方文档确认。

### 原先答案的错误
之前说 Claude Code 是"同一厨房里换厨师"。这是错的。

### 官方文档确认的事实
（来源：docs.anthropic.com/en/docs/en/sub-agents）

1. **Named subagent 默认有独立上下文隔离**
   - 官方原文："Each subagent runs in its own context window with a custom system prompt, specific tool access, and independent permissions."
   - 官方原文："A named subagent starts from its own definition." / "Context: Fresh context with the prompt you pass"
   - 官方原文："Prompt cache: Separate cache"

2. **Claude Code 的模型路由（haiku/opus）是通过 subagent frontmatter 实现的**
   - frontmatter 字段 `model`可选 sonnet/opus/haiku/inherit，默认 inherit
   - 这是在启动具有 fresh context 的独立 subagent 时指定模型，不是"同一上下文内切换模型"

3. **Fork 模式才是不隔离上下文的**
   - 官方原文："A fork is a subagent that inherits the entire conversation so far instead of starting fresh."
   - Fork 是实验性功能，需要环境变量开启
   - Named subagent 不是 fork，两者有本质区别

4. **Claude Code 自动分发机制**
   - 官方原文："Claude uses each subagent's description to decide when to delegate tasks."
   - 基于 subagent 定义文件中的 `description` 字段匹配

### 修正后的对比

| 维度 | Hermes delegate_task | Claude Code named subagent |
|------|----------------------|----------------------------|
| 触发方式 | 主模型显式调用工具 | Claude 根据 description 自动匹配，也可显式调用 |
| 上下文 | 完全隔离，只传入显式 context | Fresh context，独立 system prompt 和 tools |
| 模型选择 | acp_command 可换完全不同后端 | model frontmatter 在 Anthropic 生态内切换 |
| 文件系统 | 独立终端 session | 默认共享，可用 isolation: worktree 隔离 |
| 结果返回 | 只最终摘要返回 | 只最终结果返回 |

### 关键纠正
- Claude Code 的 named subagent 也是"隔壁独立厨房"，不是"同一厨房换厨师"。
- 两者都隔离上下文。真正差异在于：
  1. 分发决策机制（Hermes 显式工具调用 vs Claude 自动 description 匹配）
  2. 模型切换范围（Hermes 可跨后端 vs Claude 仅限 Anthropic 生态）
  3. 配置方式（Hermes JSON 参数 vs Claude YAML frontmatter）

### 教训
不能凭记忆中的"常识"回答技术架构问题，用户质疑时必须查官方文档确认。Claude Code 的 subagent 机制在近期有较大变化，过时的经验会导致严重误导。
