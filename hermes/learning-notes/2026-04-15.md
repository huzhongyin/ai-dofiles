# 2026-04-15 学习笔记

## Skills 体系架构

### 问题
Skills 是怎么引入的？bundled skills 在源码的什么位置？为什么不放到 Skills Hub (official) 里？

### 答案
Skills 有两个来源，分发策略不同：

- **Bundled** — 随源码在 `~/.hermes/hermes-agent/skills/` 目录（78 个），安装时通过 `tools/skills_sync.py` 同步到 `~/.hermes/skills/`。离线可用，跟着 `hermes update` 走。
- **Official/Hub** — 远程注册表，`hermes skills install` 手动安装，可单独更新，需联网。

类比：Bundled 是手机预装 App，Hub 是应用商店。

### 关键点
- 源码位置: `~/.hermes/hermes-agent/skills/`（git 管理）
- 运行位置: `~/.hermes/skills/`（实际使用的目录）
- 同步机制: manifest 文件 `.bundled_manifest` 记录每个 skill 的 hash
- 同步规则: 用户改过的 skill（hash 不一致）不会被源码更新覆盖

## Skills 加载时机

### 问题
Skills 什么时候被加载到系统提示词中？

### 答案
采用**渐进式披露（progressive disclosure）**：

1. **会话启动时** — `build_skills_system_prompt()` 扫描 `~/.hermes/skills/` 下所有 SKILL.md
2. **只注入摘要** — name + description（很短），拼成 `<available_skills>` 列表放进系统提示词
3. **按需加载全文** — agent 判断任务相关时，用 `skill_view(name)` 读取完整 SKILL.md 内容

### 关键点
- 每轮对话都能看到 80 个 skill 的 name+description
- 全文只在需要时才加载，节省 token
- Agent 需要自己判断"这个场景该不该 load 这个 skill"——这是个判断环节，存在漏判可能

## Memory 机制

### 问题
Memory 的存储路径是什么？什么时候加载？

### 答案
- `~/.hermes/memories/MEMORY.md` — agent 的笔记（环境事实、工具经验等）
- `~/.hermes/memories/USER.md` — 用户画像

### 加载时机
1. **会话启动** — `load_from_disk()` 读取两个 md 文件
2. **冻结快照** — 渲染成带百分比的格式块，存入内存
3. **每轮 API 调用** — 从快照注入系统提示词（不重新读磁盘）
4. **上下文压缩后** — 重新从磁盘加载（防止压缩丢失上下文）

### 关键点
- Memory 是直接生效的背景知识，不需要 agent 判断是否加载
- 用 `memory` 工具写入时同时更新磁盘 + 快照，下轮立即可见
- 字符上限: MEMORY.md 2200 字符，USER.md 1375 字符

## Memory vs Skill 的选择

### 问题
一个行为习惯（比如"学习型问题要记笔记"）应该记到 memory 还是做成 skill？

### 答案
核心区别在于 **是否需要 agent 做判断**：

| | Memory | Skill |
|---|---|---|
| 生效方式 | 直接作为背景知识 | 需要 agent 判断是否加载 |
| 可靠性 | 高（无需判断） | 中（依赖匹配判断） |
| 详细程度 | 只能写几句话 | 可以写完整规范 |
| 分享性 | 不可分享 | 可分享给团队 |

### 最佳实践：混合使用
- **Memory 存触发规则**（一句话）："用户问学习型问题时，保存笔记到 learning-notes/，格式参考 skill:learning-notes"
- **Skill 存详细规范**（格式模板、触发标准、示例）

这样 memory 保证不漏，skill 保证知道怎么做。

## 飞书集成（个人版）

### 问题
个人版飞书如何给组织中其他人发消息？通讯录 API 能搜到人吗？

### 答案
个人版飞书**没有组织架构**，通讯录 API 基本不可用：

- `GET /contact/v3/users?department_id=0` → 40004 错误（无部门概念）
- `GET /search/v1/user?query=xxx` → 99991663 错误（需要 user_access_token，个人版拿不到）
- `GET /contact/v3/users` → 只返回跟机器人说过话的人

### 找人的唯一可靠方式
让对方先给机器人发消息 → 从 gateway.log 拿 chat_id 和 open_id → 保存到 memory → 以后直接用

### 关键点
- 飞书有两层权限：API 权限开关 + 应用可用范围
- 改了可用范围后必须重新发布应用
- 常见错误码：40004（无部门权限）、230013（用户不在可用范围）、99991663（搜索API认证问题）

## Skill 同步机制的教训

### 问题
为什么 `~/.hermes/skills/` 下的 bundled skill 不应该直接编辑？

### 答案
`skills_sync.py` 用 manifest（`.bundled_manifest`）追踪每个 skill 的 hash。直接编辑会导致 hash 不一致，被标记为 user-modified，以后源码更新时**不会自动覆盖**。

### 关键点
- Manifest 格式: `skill_name:origin_hash`
- 想改 skill 用 `skill_manage(action='create')` 创建新的，不要编辑 bundled 的
- 今天就踩了这个坑：把飞书信息写进了 hermes-agent skill，导致 user-modified 标记

## 飞书发消息的 Python 参数顺序 Bug

### 问题
用 subprocess 调用 `feishu_send.py` 时，消息总是发到默认 chat_id，`--chat-id` 参数被忽略。

### 根因
`feishu_send.py` **没有 `--message` 参数**。它的消息是通过位置参数传入的。

错误调用：
```python
cmd = ["python3", script, "--message", "text", "--chat-id", chat_id]
```
脚本的参数解析器遇到 `--message`（不认识的 flag）就停止解析，把后面所有内容（包括 `--chat-id`）都当作消息文本。所以 `--chat-id` 从未被解析，走的是默认值。

正确调用：
```python
cmd = ["python3", script, "--chat-id", chat_id, "--", text]
```
`--` 分隔符明确告诉解析器：前面是 flags，后面是位置参数。

### 关键点
- CLI 工具的参数解析顺序很重要：不认识的 token 会中断 flag 解析
- 用 `--` 分隔符是 CLI 惯例，防止歧义
- 这种 bug 很隐蔽：不会报错，只是悄悄用了默认值

## GitHub Trending HTML 解析的坑

### 问题
抓取 `github.com/trending` 页面时，`<a>` 标签里的 `href` 不在最前面。

### 原因
GitHub 的 `<a>` 标签有大量 `data-hydro-click` 等属性在 `href` 之前。如果正则写成 `<a\s+href="..."` 就匹配不到。

正确正则：`<a[^>]+href="/([^/]+/[^/"]+)"`
错误正则：`<a\s+href="/([^"]+)"`（匹配不到，因为 href 前面有其他属性）

### 关键点
- 解析生产环境 HTML 时，永远不要假设属性顺序
- `[^>]+` 比 `\s+` 更稳健（匹配任意属性而非仅空白）


## Skill 触发机制的根因分析

### 问题
内置了 software-development skills（writing-plans、TDD、systematic-debugging），为什么实现需求时没有用？如何确保新会话准确触发？

### 答案
Skills 加载依赖 LLM 主动扫描系统提示词中的 available_skills 列表。没有结构化的强制门禁机制，全靠"Before replying, scan... you MUST load"这条自然语言指令。

没触发的根本原因：
1. Skills 格式是"参考列表"，不是"工作流门禁" — LLM 将其视为背景知识，不是执行前必读流程
2. "I already know this" 覆盖了 scan 指令 — 当自信能完成任务时，跳过了 skill 扫描
3. SKILL.md 里的触发条件藏在内部 — 不加载就看不到完整的触发标准
4. 没有 task classification 中间环节 — 思维链是"需求→执行"，缺少"需求→分类→匹配 skill→执行"

### 关键点
- 这是 LLM 行为层面问题，不是框架设计缺陷
- 正确做法：在 memory 中写入强制规则，每轮对话前做 task classification

## Memory 设计原理

### 问题
Memory 为什么这么容易就满了？满了怎么办？删除的记录去哪了？

### 答案
Memory 刻意很小 — MEMORY.md 2200字符（~800 tokens），USER.md 1375字符。每轮注入系统提示词，占 context window。

满了直接拒绝写入，不会自动压缩。删除后永久消失，没有回收站/版本历史/undo。

多层记忆架构：
- Hot: MEMORY.md（每轮注入，存最高优先级事实）
- Warm: Skills（按需加载的工作流）
- Cold: Session transcripts（通过 session_search 搜索）

### 关键点
- 定期精简 memory，非核心内容移到 skill 或 session_search
- replace 比 remove 安全（至少内容还在）

## state.db 的真实用途

### 问题
hermes 的 state.db 是不是用来存历史 memory 的？

### 答案
不是。state.db 是 session_search 的底层存储，存的是对话转录，不是 memory 历史。

表结构：sessions（会话元数据）、messages（每轮完整内容）、messages_fts（全文搜索索引）。

当前 539 条 messages 记录了所有对话轮次。删除 memory 后，只有通过 session_search 搜 messages 表才能找回。

### 关键点
- 没有 memory_history 表
- state.db = session transcript store，不是 memory version store

## 注意力残差（Attention Residual）
- 来源：认知心理学 / 行为经济学
- 核心：注意力切换任务后不会立刻归零，有"残留"粘在上一件事上
- 恢复时间：平均需要约23分钟完全恢复深度专注
- 影响：深度工作被碎片化，频繁切换导致效率骤降
- 设计陷阱：短视频/社交媒体利用此原理制造粘性
- 类比：水龙头关了之后还会滴几滴

## 通缩死亡螺旋（Deflationary Death Spiral）
- 来源：宏观经济学
- 核心：物价下跌→预期降价→推迟消费→企业裁员→收入下降→物价进一步下跌（自我强化的恶性循环）
- 关键机制：
  - 债务通缩（费雪效应）：物价跌，债务实际负担加重
  - 零利率陷阱：传统货币政策失效
  - 预期固化：行为模式难以扭转
- 真实案例：日本失去的三十年、美国大萧条
- 一句话：越便宜越不买，越不买越便宜，直到经济崩盘

## 两者的隐喻联系
- 共同本质：资源（注意力/货币）应该流动但被卡住了
- 注意力残差→越切换越低效；通缩螺旋→越通缩越不敢花
