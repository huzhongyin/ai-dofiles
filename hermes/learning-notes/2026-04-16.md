## GitHub Trending HTML 总 Star 数正则修复

### 问题
scrape_github_trending 中总 star 数的正则 `r'class="Link[^\s]*\s+muted[^"]*"[^>]*>\s*(?:<svg[^>]*>.*?</svg>)?\s*([\d,]+)'` 无法匹配，导致所有项目 stars=0。

### 答案
GitHub Trending 页面的 star 数格式是 `</svg>\n        44,128</a>`，即 SVG star 图标后紧跟数字。正确正则：`r'</svg>\s*([\d,]+)</a>'`

### 关键点
- GitHub Trending 页面结构可能随时间变化，需要定期检查正则
- 飞书不渲染 markdown 表格，用纯文本对齐 + █▓ 条形图替代

## 飞书消息格式优化

### 问题
报告输出太粗糙，没有"肉眼一眼可以观察出来的效果"。

### 答案
从 markdown 表格改为纯文本对齐格式：
- 用 ▓ 条形图可视化 star 增长（按最大值比例，最多 8 格）
- 用 █ 条形图显示领域热度分布
- 固定列宽对齐，飞书/终端都能完美显示
- 分类 tag 最多 8 字符，避免截断有意义的词
- 排名变化用 NEW/↑/↓/▲/▼ 符号

### 关键点
- 不要依赖 markdown 渲染（飞书、Telegram 等支持不一）
- 纯文本对齐 + Unicode 字符（█▓）是最可靠的跨平台可视化方式

## Hermes 日志与 HTTP 请求观察手段

### 问题
如何观察 Hermes Agent 交互过程中的 HTTP 请求和 skill 加载流程？

### 答案

**1. Session JSON 文件（最直接）**
路径：`~/.hermes/sessions/session_*.json`
包含：system_prompt（含 available_skills 索引）、每条 assistant 消息的 tool_calls、tool 消息的返回内容、reasoning 字段。

**2. Verbose 模式**
命令：`hermes -v`
日志到：`~/.hermes/logs/agent.log`
记录：API 请求详情（模型/消息数/token估算）、tool call 完整参数、tool 结果完整内容、token 使用量、API 响应时间。

**3. HTTP 请求 Dump（错误时触发）**
`_dump_api_request_debug()` 在 API 出错时自动 dump 到 `~/.hermes/logs/api_debug_<timestamp>.json`。
包含完整 HTTP body：model、messages（含 system prompt）、tools schema、headers。

**4. 客户端抓包**
```bash
# httpx 详细日志
export HTTPX_LOG_LEVEL=debug

# 或 mitmproxy
mitmdump --listen-port 8080
export HTTPS_PROXY=http://127.0.0.1:8080
```

**5. Skill 索引缓存**
`cat ~/.hermes/.skills_prompt_snapshot.json` — 包含所有 skill 元数据、mtime/size manifest。

### 关键点
- Session JSON 是最方便的分析入口，包含完整的消息流
- `-v` flag 开启后会在 agent.log 记录详细的 API 调用过程
- HTTP body dump 只在 API 出错时触发，正常运行不产生
- 客户端抓包可以看到真实的网络层 HTTP 请求/响应

## Claude Code 指令/Skill 加载机制分析

### 问题
Claude Code 如何加载指令和 skill？交互过程和日志查询方式是什么？

### 答案

**Claude Code 采用「直接注入」架构，与 Hermes 的「渐进式披露」截然不同。**

#### 1. 指令加载：CLAUDE.md 直接注入 System Prompt

Claude Code 不用 skill 索引 + tool_call 的两阶段模式，而是把 CLAUDE.md 的**完整内容直接拼进 system prompt**：

- `~/.claude/CLAUDE.md` — 全局指令（所有项目生效）
- `./CLAUDE.md` — 项目级指令（git-tracked，团队共享）
- `.claude/CLAUDE.local.md` — 本地个人指令（gitignored）
- `.claude/rules/*.md` — 模块化规则文件（替代单个巨型 CLAUDE.md）

每次会话启动时，这些文件的**完整内容**直接注入 system prompt。没有索引、没有 tool_call、没有按需加载。

#### 2. Skill 加载：也是直接注入，靠自然语言触发

Claude Code 的 skills 存放在 `~/.claude/skills/` 或 `.claude/skills/` 目录，每个 skill 是一个 SKILL.md 文件。

与 Hermes 不同：
- **没有 skill 索引**：不会在 system prompt 里列出所有 skill 的 name + description
- **没有 skill_view tool**：不通过 tool_call 按需加载
- **直接注入完整内容**：所有 SKILL.md 的完整内容在会话启动时就注入 system prompt
- **触发方式**：靠 SKILL.md 的 frontmatter 中的 `description` 字段描述触发条件，模型在收到用户请求时自然判断是否匹配

skill-log.md 记录了每次 skill 调用：
```
### 10:15 | BUG_FIX → log-analyzer-optimized
**触发意图：** BUG_FIX（分析自动闭锁失败测试问题）
**调用对象：** Skill `log-analyzer-optimized`
**目的：** 通过日志分析定位自动闭锁功能失效的根本原因
```

#### 3. Slash Commands：也是直接注入

`.claude/commands/*.md` 定义的自定义斜杠命令，完整内容也在启动时注入。用户输入 `/command-name` 触发。

#### 4. 关键差异对比

| 维度 | Hermes | Claude Code |
|------|--------|-------------|
| 架构 | 渐进式披露（Progressive Disclosure） | 直接注入（Direct Injection） |
| 索引 | system prompt 中注入 83 个 skill 的 name + description | 无索引 |
| 完整内容 | 通过 skill_view(name) tool_call 按需加载 | 会话启动时全部注入 system prompt |
| Token 成本 | 索引 ~11K chars，完整内容按需 | 所有 skill 完整内容一次性注入 |
| 触发方式 | system prompt 指令 "MUST load with skill_view(name)" | 自然语言匹配 SKILL.md description |
| 缓存 | snapshot 缓存 (mtime/size manifest) | 无特殊缓存，依赖 LLM prompt caching |
| Context 占用 | 索引固定占位，内容按需 | 所有内容固定占位 |

#### 5. 日志和调试方式

**Claude Code 日志体系：**
- `~/.claude/skill-log.md` — Skill 调用日志（48KB），记录每次 skill 触发的意图、调用对象、目的、结果
- `~/.claude/history.jsonl` — 命令历史（时间戳 + prompt）
- `~/.claude/session-env/` — 会话环境变量快照（368 个会话）
- `~/.claude/projects/<project>/memory/` — 自动记忆（每项目 25KB/200 行限制）
- `~/.claude/debug/` — 调试日志（需 `-d` 或 `--debug-file` 开启）
- `~/.claude/file-history/` — 文件编辑历史（104 个文件快照）

**开启调试：**
```bash
# 基础调试
claude -d

# 指定类别
claude -d "api,hooks"

# 写入文件
claude --debug-file /tmp/claude-debug.log

# Stream JSON 输出（最详细的结构化日志）
claude -p "task" --output-format stream-json --verbose --include-partial-messages
```

**Stream JSON 事件类型：**
- `system` — 系统消息
- `assistant` — 助手回复（含 tool_calls）
- `user` — 用户消息
- `result` — 最终结果（含 session_id, num_turns, total_cost_usd, usage）

**查看 HTTP 请求：**
- Claude Code 不直接暴露 HTTP 请求/响应
- 可以通过 `--debug-file` + api 类别获取 API 调用详情
- 或设置代理：`ANTHROPIC_BASE_URL` 指向本地代理

#### 6. Session 文件对比

| | Hermes | Claude Code |
|---|--------|-------------|
| 格式 | 单个 JSON 文件 | JSONL（每行一个事件） |
| 路径 | `~/.hermes/sessions/session_*.json` | `~/.claude/sessions/` (空，可能在其他位置) |
| 内容 | system_prompt + messages + tools | 事件流（system/user/assistant/result） |
| 保存 | 自动持久化 | 需要会话持久化开启（默认开） |

### 关键点
- Claude Code 的「直接注入」更简单，但 token 成本更高（所有 skill 内容都占 context）
- Hermes 的「渐进式披露」更节省 token，但增加了 tool_call 的交互轮次
- Claude Code 的 SKILL.md description 类似 Hermes 的 skill 索引 description，但前者是注入 system prompt 的一部分，后者是指导模型何时调用 skill_view
- Claude Code 的 skill-log.md 是 Hermes 没有的——它自动记录每次 skill 调用的元数据

## Claude Code Skill 加载机制（修正版）

### 问题
Claude Code 是全量注入还是渐进式披露？

### 答案
Claude Code **也是渐进式披露**，和 Hermes 同构。我之前的分析错了。

#### 实际机制（从 debug 日志和 using-superpowers SKILL.md 验证）

**第一步：会话启动时发送 skill 元数据**
- Debug 日志：`Sending 92 skills via attachment (initial, 92 total sent)`
- 92 个 skill 的 name + description 通过 Anthropic API 的 `attachments` 机制发送
- 每个 skill 的 metadata 被标注为 `Skill prompt: showing "plugin:skill-name"`
- 这一步和 Hermes 的 `<available_skills>` 索引等价

**第二步：模型通过 `Skill` tool 按需加载完整内容**
- using-superpowers SKILL.md 明确说：`Use the Skill tool. When you invoke a skill, its content is loaded and presented to you—follow it directly. Never use the Read tool on skill files.`
- 模型扫描 92 个 skill 的 name + description，判断哪些相关
- 通过 `Skill` tool_call 加载完整 SKILL.md 内容
- 这一步和 Hermes 的 `skill_view(name)` tool_call 等价

#### 关键差异（Claude Code vs Hermes）

| 维度 | Claude Code | Hermes |
|------|-------------|--------|
| 架构 | 渐进式披露 ✓ | 渐进式披露 ✓ |
| 元数据传递方式 | Anthropic API `attachments` 机制 | system prompt 中的 `<available_skills>` XML 块 |
| 加载工具 | `Skill` tool (Anthropic 原生) | `skill_view` tool (自定义) |
| Skill 数量 | 92 个 | 83 个 |
| 系统指令 | using-superpowers SKILL.md 注入 | system prompt 中的 "## Skills (mandatory)" |
| 强制程度 | "IF A SKILL APPLIES, YOU MUST USE IT" | "you MUST load it with skill_view(name)" |
| 触发方式 | 同上：name + description 匹配 | 同上：name + description 匹配 |
| 缓存 | 无特殊缓存（依赖 API prompt caching） | snapshot 缓存 (mtime/size manifest) |

#### 架构等价关系

```
Claude Code:                          Hermes:
attachments: [                         system prompt:
  {name: "tdd", desc: "..."},          <available_skills>
  {name: "debug", desc: "..."},          - tdd: ...
  ...92 个...                            - debug: ...
]                                      </available_skills>

     ↓ Skill tool_call                      ↓ skill_view tool_call
     
Skill("tdd") → 返回完整内容           skill_view("tdd") → 返回完整内容
```

两者都是：**索引（metadata）先给模型看 → 模型判断命中 → tool_call 加载完整内容**。

### 关键点
- 两者都是渐进式披露，不是全量注入
- 差异在于传递 metadata 的渠道：Claude Code 用 API attachments，Hermes 用 system prompt
- Claude Code 的 `Skill` tool 是 Anthropic API 原生支持的，Hermes 的 `skill_view` 是自定义的 OpenAI-format tool
- using-superpowers SKILL.md 的作用类似于 Hermes system prompt 中的 "## Skills (mandatory)" 段

### 关键点
- Claude Code: `claude -d` → 看 `~/.claude/debug/*.txt` → 关注 "Sending N skills via attachment" 和 "Skill prompt: showing" 日志
- Hermes: `hermes -v` → 看 session JSON → 关注 skill_view tool_calls

## Hermes 工具权限配置（edit/read/bash）

### 问题
Hermes 如何配置 edit、read、bash 等权限？

### 答案
Hermes 按 **toolset（工具集）** 整体开关，不支持细粒度到"只允许 read 不允许 edit"。

**能力与 toolset 对应：**
- read/edit → `file` toolset（read_file, write_file, patch, search_files）
- bash → `terminal` toolset（terminal shell 命令）
- 代码执行 → `code_execution` toolset（execute_code 沙箱 Python）

**配置方式：**
1. `hermes tools` — 交互式 curses UI 开关
2. `hermes tools enable/disable <name>` — 命令行开关
3. `config.yaml` 的 `platform_toolsets` — 按平台分别配置
4. `approvals.mode: manual` — 危险命令需手动确认
5. `command_allowlist` — 限制 terminal 只能执行白名单命令
6. `security.tirith_enabled` — 安全扫描

**修改后需 `/reset`（新会话）才生效。**

### 关键点
- 最小控制单位是 toolset 整个开关，不能单独控制 read vs edit
- `command_allowlist` 可以限制 bash 能执行的命令
- `approvals.mode: manual` 实现逐命令审批
- CLI 模式 `--yolo` 跳过所有审批
