方案书生成必须使用**调用链驱动**方法，严禁全量文件搜索。

**Why:** 用户在 auto-lock 案例中演示了专家级排查路径，发现之前的 A-F 穷举式方法（grep所有文件→全量阅读）效率低且不严谨，遗漏了多态解析和配置追溯。

**How to apply:**
1. 从日志关键字出发定位入口方法，不是搜索所有包含关键词的文件
2. 遇到基类/子类关系时，必须追踪初始化逻辑（如通过 `isBleLocationVersionV1` 确认使用哪个 Handler）
3. 配置值无法从代码确定时，暂停询问用户/模块负责人
4. 条件必须区分三层：前提条件（进入流程前）/ 激活条件（触发核心行为）/ 执行条件（激活后判断）
5. 激活前的所有失效路径记为前提条件，激活后的判断记为执行条件
6. 到达最终指令发送（网络请求/BLE指令）才算正常执行流
7. 沿调用链路径收集日志关键字，而非独立步骤全量 grep
8. **Step 2B 强制门控**：方案书不存在时，必须立即暂停并要求用户提供入口信息（日志关键字/类名/方法名），严禁自行搜索代码库推断入口 — 原因：自行搜索可能选错入口模块，且关键字来源无用户验证，结果不可靠
