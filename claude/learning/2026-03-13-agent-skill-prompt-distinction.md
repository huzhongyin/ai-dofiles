# Agent / Skill / prompt.md 使用场景区分

**日期：** 2026-03-13
**意图类型：** LEARNING
**核心问题：** 三者在使用场景上的严格区分是什么？

---

## 用户初始理解（评估基准）

- Agent 具备自主能力，可以发散 ✅
- Skill 是懒加载、语义匹配触发，省 token，最小上下文原子性 ✅
- prompt.md 是固定模板，不具备发散性和创造力 ⚠️（部分准确）
- 三者目的都一样，让模型更好理解上下文 ❌（根本误区）

---

## 核心纠正

三者的区别**不是上下文管理策略**，而是**执行权限和自主程度**：

```
prompt.md   ──→  【读】  纯文本注入，无工具调用
skill       ──→  【读+工具】  可调用工具，流程预定义
agent       ──→  【读+工具+子进程】  独立执行循环，可 spawn 子 agent
```

---

## 精确对比表

| 维度 | prompt.md | Skill | Agent |
|------|-----------|-------|-------|
| 存在形式 | Markdown 文件 | 带指令的 Markdown | 独立进程/subprocess |
| 工具调用 | ❌ 无 | ✅ 有（Read/Write/Bash 等） | ✅ 有（全部工具） |
| 触发方式 | 自动加载 / 手动引用 | 语义匹配 / slash command | 显式调用 Agent tool |
| 执行循环 | ❌ 无 | ❌ 无（单次执行） | ✅ 有（多轮 agentic loop） |
| 上下文隔离 | ❌ 共享主上下文 | ⚠️ 部分共享 | ✅ 独立上下文 |
| 适合场景 | 规范约束、行为设定 | 标准化操作流程 | 复杂多步任务 |

---

## 一句话区分

- **prompt.md** = 告诉模型「你应该是谁 / 要遵守什么」（身份+约束）
- **Skill** = 告诉模型「遇到这类事情，按这个 SOP 走」（流程封装）
- **Agent** = 让一个独立的模型实例「去完成这个任务，自己想办法」（委托执行）

---

## 类比记忆

想象你是技术负责人：
- **prompt.md** = 《行为规范手册》，员工上班前读一遍
- **Skill** = 《标准操作流程卡》，遇到某类问题按卡片执行
- **Agent** = 外包给另一个人，他有自己的电脑和工作区，完成后汇报

---

## Skill 的补充说明

"省 token" 只是 Skill 的副产品，核心价值是：
**把固定的「操作流程」封装成可复用命令**，使标准化操作可以按需触发、原子执行。
